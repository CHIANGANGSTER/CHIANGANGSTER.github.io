实战路径：从素材到可交付画面

General ComfyUI Production Path

ComfyUI 的优势不是“点一次生成一张图”，而是把图像生产拆成可检查、可替换、可复用的节点链路。对平面设计来说，它更像一套视觉生产台：输入素材，明确画面目标，控制模型和参数，最后把结果整理成可交付资产。

明确输入 / Input First

先判断任务需要什么输入：产品白底图、参考图、深度图、线稿、遮罩、品牌色或场景关键词。输入越清楚，后面的节点越容易稳定复现。

拆分目标 / Split the Job

把需求拆成结构、材质、光线、风格和输出尺寸。不要把所有要求塞进一个提示词里，能用节点控制的就交给节点，能用提示词描述的再写进提示词。

小图验证 / Preview Before Pressure

先用低成本预览确认构图、透视和风格方向，再开启高清修复、放大、细节增强或批量生成。这样能减少无效等待，也方便快速定位问题来自哪一段节点。

锁定版本 / Save the Workflow

交付前保存工作流、关键提示词、模型版本、种子和输出图。ComfyUI 的价值在于“下次还能做出来”，不是只保留最后一张图片。

ComfyUI 常见问题自查手册

General Troubleshooting Guide

排查 ComfyUI 问题时，不要一上来就怀疑模型。先看输入、节点连接、模型路径、显存和参数范围。大多数问题都能通过“从输出往前倒查”的方式快速定位。

Q: 节点报红、缺节点或工作流打不开？

常见原因：缺少自定义节点、节点版本不一致，或别人导出的工作流使用了你本机没有安装的扩展。
处理方式：先看报错节点名称，再安装对应 custom node；安装后重启 ComfyUI；如果仍然报错，再检查节点输入口是否被新版节点改名。

Q: 模型、LoRA、VAE 明明下载了，却在节点里找不到？

常见原因：文件放错目录、模型命名不规范，或 ComfyUI 没有重新扫描。
处理方式：确认 checkpoint、loras、vae、controlnet 等文件放在对应 models 子目录；文件名尽量保留模型版本信息；刷新或重启后再检查节点下拉列表。

Q: 生成很慢、卡住，或者显存爆掉？

常见原因：分辨率过大、批量数过高、同时开启高清修复/放大/ControlNet，或后台占用显存。
处理方式：先降低分辨率和 batch size；预览阶段关闭放大和细节增强；确认效果后再分段加压。复杂工作流建议先跑小图，最后再跑交付尺寸。

Q: 画面跑偏，产品结构变形，或者风格不稳定？

常见原因：提示词目标混乱、参考图权重过强或过弱、种子频繁变化，导致模型每次都重新理解任务。
处理方式：固定 seed，先锁定构图和产品结构，再逐步调整风格词；涉及产品图时优先保护轮廓、比例和材质边界，不要同时大幅修改空间、灯光和材质。

Q: 输出图看起来“AI 味”重，细节脏或边缘发虚？

常见原因：采样步数、降噪强度、放大链路和提示词细节没有配合好。
处理方式：把修复拆成两步：先解决大结构，再解决材质和边缘；放大时降低 denoise，避免把原有细节重画掉；必要时用遮罩局部修复，不要整图反复重绘。

ComfyUI 工作流管理建议

Workflow Hygiene for Design Teams

节点命名

重要节点用中文或英文短名标注用途，例如“产品输入”“材质增强”“最终输出”。不要只保留默认节点名，否则多人协作时很难维护。

分组管理

把输入、控制、采样、修复、放大、输出分成不同 Group。复杂工作流先看分组，再看节点，排查效率会高很多。

版本记录

每次稳定出图后保留工作流 JSON、关键提示词、模型版本和示例输出。ComfyUI 支持工作流随图片保存，适合沉淀团队案例。

API 思维

当一个流程稳定后，可以把它整理成可调用的生产流程，用统一输入替换手动节点操作，为批量图像生产和内部工具打基础。
